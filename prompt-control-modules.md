# Prompt Control Modules

这份文档的目标，是把原本分散在提示词手册中的控制指令库，升级成一套真正可插拔、可映射、可自动注入的控制模块层。它不再只是若干好用句子的集合，而是一套面向 Prompt System 的控制部件库。你可以把它理解成 Prompt 的中间层：上层是用户需求，下层是最终成品 Prompt，而这一层负责决定应该往 Prompt 里注入哪些控制结构。

如果 `prompt-playbook-quick-reference.md` 解决的是“怎么快速用”，`prompt-playbook-examples.md` 解决的是“单句需求如何生成成品 Prompt”，那么这份文档解决的就是“系统到底往 Prompt 里注入了什么，以及为什么这样注入”。它是整套 Prompt System 里最接近编译器内部规则的一层。

## 一 模块层的作用

控制模块层的意义，不是让 Prompt 变长，而是让 Prompt 的控制职责更清晰。用户给出的需求通常是表层输入，而高质量 Prompt 所需要的却是结构性控制信息，例如问题是否需要重定义、是否需要批判性、是否必须收敛到行动、是否应该重点压制模板腔。控制模块层就是用来补齐这些信息的。

一个成熟的 Prompt 生成系统，不应当直接从需求跳到成品 Prompt，而应当先经过三个动作。第一，识别任务类型。第二，识别高风险失败模式。第三，从控制模块层里选取最合适的模块进行注入。只有这样，系统的输出质量才是可解释、可调试、可迁移的。

## 二 模块的标准结构

每个控制模块都应当具备统一结构。这样才能被系统使用，而不是只能被人类凭感觉拼接。

```text
模块名称：
模块目标：
主要修复的失败模式：
适用任务：
推荐插入位置：
短版指令：
长版指令：
常见收益：
可能副作用：
使用备注：
```

下面开始正式定义这套模块库。

## 三 模块 A：问题重定义模块

```text
模块名称：问题重定义
模块目标：把用户表层描述翻译成更高质量的问题定义
主要修复的失败模式：表层复述、面面俱到无判断、任务理解过浅
适用任务：算法分析、源码分析、架构规格、商业洞察、A 股分析、通用深度分析
推荐插入位置：角色层后，认知路径层前
短版指令：用户的描述是起点，不是边界。先识别这个问题真正属于什么，再组织回答。
长版指令：不要顺着用户的表面表述直接作答。先重述问题，区分表象与本质，识别隐含需求、底层目标和真正的求解对象，再基于那个层级组织回答。
常见收益：显著压制复述型输出，提高结构判断能力
可能副作用：在极简单任务中可能显得过重
使用备注：几乎适用于所有复杂任务，是默认优先注入模块之一
```

## 四 模块 B：认知下钻模块

```text
模块名称：认知下钻
模块目标：强迫模型经过关键分析台阶，而不是直接给模板答案
主要修复的失败模式：表层解释、伪深度、正确废话
适用任务：算法分析、源码分析、商业洞察、通用深度分析、复杂写作分析
推荐插入位置：问题重定义层后，质量门槛层前
短版指令：不要停留在表层解释，继续分析底层结构、关键机制和决定结果的因素。
长版指令：不要只解释现象或功能。继续追问它解决了什么根问题、避免了什么代价、依赖了什么约束、哪些看起来次要却实际上决定质量上限。
常见收益：提升解释力、机制感和思维密度
可能副作用：在要求极简回答时可能拉长输出
使用备注：适合用户要求“深度分析”“高水平”“讲透”时自动注入
```

## 五 模块 C：关键点优先模块

```text
模块名称：关键点优先
模块目标：避免平均展开，聚焦真正决定结果上限的少数关键因素
主要修复的失败模式：面面俱到无判断、正确废话、信息分散
适用任务：商业洞察、A 股分析、架构规格、算法分析、通用深度分析
推荐插入位置：认知路径层或质量门槛层
短版指令：不要平均分配篇幅，优先分析真正决定结果质量上限的少数关键点。
长版指令：不要把所有方面都讲一点。优先指出最关键但最容易被忽略的因素，并解释为什么它比其他点更重要。如果某个部分删掉后不影响理解，应压缩其篇幅。
常见收益：显著提升判断力和文章锋利度
可能副作用：在要求“全面梳理”的任务中可能造成覆盖不足
使用备注：当用户需求包含“重点”“关键”“核心”“为什么重要”时优先注入
```

## 六 模块 D：批判性模块

```text
模块名称：批判性
模块目标：让模型不仅解释为什么成立，也分析代价、局限和脆弱点
主要修复的失败模式：无批判赞美、伪专业、无权衡分析
适用任务：源码分析、架构规格、商业洞察、A 股分析、评审类任务
推荐插入位置：认知路径层或质量门槛层
短版指令：不要默认现有方案、代码或观点天然合理，要分析它的收益、代价、局限和替代路径。
长版指令：对于关键实现或结论，不要只分析它为什么成立，还要分析它可能带来的代价、脆弱点、失效方式以及在什么条件下应当考虑替代方案。不要无条件赞美，也不要无差别中立。
常见收益：提高可信度、真实判断力和评审质量
可能副作用：在单纯鼓励型写作任务中可能显得过于冷峻
使用备注：评审、分析、决策类任务应高频使用
```

## 七 模块 E：信息密度模块

```text
模块名称：信息密度
模块目标：压缩常识铺垫和正确废话，让每段都提供有效增量
主要修复的失败模式：正确废话、空话连篇、结构完整但无洞察
适用任务：所有深度分析任务、写作任务、商业/投资/架构分析
推荐插入位置：质量门槛层
短版指令：每一段都必须提供新的判断、约束、洞察或区分，不要用常识性铺垫稀释内容。
长版指令：删除不影响理解的句子，压缩正确但无信息增量的表达。宁可少讲一些，也不要用面面俱到和套话填充篇幅。每一段都应推动理解向前，而不是原地打转。
常见收益：提高阅读价值和专业密度
可能副作用：在面向新手的教学内容中可能显得不够铺垫
使用备注：用户要求“高信息密度”“不要废话”“尽量高级”时强烈建议注入
```

## 八 模块 F：边界与验证模块

```text
模块名称：边界与验证
模块目标：要求关键结论具备前提、适用边界、失效条件和验证方式
主要修复的失败模式：伪深度、空泛结论、不可验证判断
适用任务：架构规格、A 股分析、商业洞察、算法分析、决策类任务
推荐插入位置：质量门槛层
短版指令：所有关键结论都必须说明前提、适用边界、失效条件和验证方式。
长版指令：不要只给结论。所有关键判断都必须说明它成立的前提、适用边界、失效条件，以及怎样验证它是否成立。如果输出涉及设计、策略或决策，还必须给出可验证的成功标准。
常见收益：显著减少伪深度和空泛判断
可能副作用：在轻量表达型任务中可能增加篇幅
使用备注：一切涉及“方案、建议、判断、预测”的任务都适合注入
```

## 九 模块 G：可执行性模块

```text
模块名称：可执行性
模块目标：让输出收敛为可行动、可实施、可决策的结果
主要修复的失败模式：不可执行、只会分析不会落地、结论不收敛
适用任务：架构规格、A 股分析、商业建议、策略建议、行动规划
推荐插入位置：质量门槛层末尾
短版指令：结论必须服务于行动，明确下一步该做什么、何时做、什么信号下调整。
长版指令：不要让分析停留在描述层。最终输出必须收敛为可执行判断，明确下一步该做什么、在什么条件下做、成功标准是什么、出现什么信号时应调整、停止或修正判断。如果建议无法实施或验证，说明答案还不完整。
常见收益：提高决策价值和现实可用性
可能副作用：在纯解释型任务中可能让结尾显得过度收敛
使用备注：只要用户的问题本质上面向行动，就应该默认注入
```

## 十 模块 H：风格控制模块

```text
模块名称：风格控制
模块目标：降低模板腔和 AI 味，让正文以自然叙述承载分析
主要修复的失败模式：模板腔、教学腔、总结腔、风格不稳
适用任务：几乎所有输出型任务，尤其是深度写作和分析性表达
推荐插入位置：全局文风层
短版指令：正文以自然、连贯的叙述性段落为主，减少模板化和 AI 腔。
长版指令：正文请以流畅、自然、连贯的叙述性段落为主。不要使用高频教学腔、总结腔和模板引导语，不要把正文写成分点堆砌或标准答案。结构化格式只用于天然适合结构化表达的部分，正文段落负责承载推理、判断和洞察。
常见收益：显著提升可读性和真实作者感
可能副作用：在需要极强结构化展示的任务里，若过度使用可能削弱扫读性
使用备注：几乎是默认注入模块，除非任务天然要求清单式输出
```

## 十一 模块组合的基本原则

模块不是越多越好。系统在注入模块时，应遵循三个原则。

第一，优先注入与任务类型最相关的模块，而不是八个全上。第二，优先注入能修复高风险失败模式的模块，而不是追求形式完整。第三，短 Prompt 也可以高质量，只要它覆盖了关键控制职责。

一个实用经验是：大多数复杂任务通常只需要四到五个模块就能形成稳定输出。超过这个数量时，边际收益会下降，且容易让 Prompt 变得臃肿。

## 十二 任务类型到模块的映射表

下面这张表是自动注入规则的核心。

```text
算法分析型
默认注入：问题重定义 + 认知下钻 + 关键点优先 + 边界与验证 + 风格控制
可选增强：信息密度

源码分析型
默认注入：问题重定义 + 认知下钻 + 批判性 + 关键点优先 + 风格控制
可选增强：信息密度 + 边界与验证

架构规格型
默认注入：问题重定义 + 批判性 + 边界与验证 + 可执行性 + 风格控制
可选增强：关键点优先

商业洞察型
默认注入：问题重定义 + 认知下钻 + 关键点优先 + 批判性 + 风格控制
可选增强：边界与验证 + 可执行性

A 股分析型
默认注入：问题重定义 + 关键点优先 + 批判性 + 边界与验证 + 可执行性 + 风格控制
可选增强：信息密度

通用深度分析型
默认注入：问题重定义 + 认知下钻 + 关键点优先 + 边界与验证 + 风格控制
可选增强：信息密度 + 批判性

写作生成型
默认注入：风格控制 + 信息密度
可选增强：问题重定义
```

## 十三 失败模式到模块的映射表

如果你不是先按任务类型注入，而是先看到坏输出再修，那么下面这张表最重要。

```text
表层复述型
-> 问题重定义 + 认知下钻

模板腔型
-> 风格控制 + 信息密度

正确废话型
-> 信息密度 + 关键点优先

伪深度型
-> 边界与验证 + 认知下钻

无批判赞美型
-> 批判性

面面俱到无判断型
-> 关键点优先 + 信息密度

不可执行型
-> 可执行性 + 边界与验证

风格不稳型
-> 风格控制
```

## 十四 自动注入规则

如果要把这套模块层真正接到“单句需求 -> 成品 Prompt”的生成器里，可以使用下面这套自动注入流程。

```text
输入：用户的一句需求或一个问题

步骤一：识别任务类型
步骤二：识别输出产物
步骤三：识别质量目标
步骤四：预判高风险失败模式
步骤五：根据任务类型加载默认模块组合
步骤六：根据失败模式追加修复模块
步骤七：根据质量目标做权重调整
步骤八：将模块转换为具体 Prompt 语句并插入五层结构
步骤九：输出成品 Prompt
```

质量目标对模块权重的影响可以进一步细化。例如：

```text
如果用户更强调深度
-> 提高 问题重定义 + 认知下钻 + 关键点优先 的优先级

如果用户更强调可执行性
-> 提高 可执行性 + 边界与验证 的优先级

如果用户更强调批判性
-> 提高 批判性 + 关键点优先 的优先级

如果用户更强调自然文风
-> 提高 风格控制 + 信息密度 的优先级

如果用户更强调结构完整
-> 提高 边界与验证 + 问题重定义 的优先级
```

## 十五 模块插入位置示意

为了让系统装配时更稳定，建议统一按下面结构插入模块。

```text
角色层
- 角色本体

问题重定义层
- 问题重定义模块

认知路径层
- 认知下钻模块
- 关键点优先模块
- 批判性模块（如适用）

质量门槛层
- 信息密度模块
- 边界与验证模块
- 可执行性模块

全局文风层
- 风格控制模块
```

这样做的好处是，系统不会把所有指令混成一段大杂烩，而是始终知道每句话在 Prompt 中承担什么职责。

## 十六 给生成器使用的最简模块装配模板

如果你要把这套模块库直接接入生成器，可以使用下面这个中间模板。

```text
基于用户需求，先判断任务类型、输出产物、质量目标和高风险失败模式。
然后从控制模块库中选择最适合的模块组合。
再将这些模块装配进以下 Prompt 结构：

1. 角色层
2. 问题重定义层
3. 认知路径层
4. 质量门槛层
5. 全局文风层

最后输出一份可直接复制使用的完整 Prompt。
```

## 十七 最后原则

控制模块层真正的价值，不是让你拥有更多句子，而是让系统知道应该在什么场景下补什么结构。只有当控制指令从“经验句子”升级成“可映射、可组合、可自动注入的模块”时，Prompt System 才真正从手工艺进入工程化。

如果要把这份文档压缩成一句话，那就是：Prompt 的质量，不只取决于你写了什么，更取决于你是否知道该在什么任务里注入什么控制模块。
