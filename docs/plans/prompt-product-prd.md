# Prompt Product PRD

## 一 产品名称

暂定名称：Prompt Studio

这是一个独立的 Prompt 产品，不依附于任何特定垂直业务系统。它的核心目标不是完成某个具体行业任务，而是帮助用户把自然语言需求稳定转化为可直接使用的高质量 Prompt，并支持 Prompt 的生成、调试、评估与沉淀。

这个产品首先服务的，并不是已经擅长写 Prompt 的用户，而是那些在陌生领域面前不知道如何使用精确语言、专业术语或结构化表达来描述需求的普通用户。用户真正的痛点往往不是“没有想法”，而是“有想法却无法用模型最容易理解的方式表达出来”。因此，Prompt Studio 的核心任务不是要求用户先学会专家表达，而是主动替用户完成需求翻译、结构补全和表达增强。

## 二 产品定位

Prompt Studio 是一个 Prompt Compiler，而不是一个聊天工具、模板仓库或内容生成器。

它的最小心智模型应当非常清晰：

- 用户输入一句需求、一个问题或一个模糊目标
- 系统自动理解任务类型与输出意图
- 系统自动补齐 Prompt 所需的控制结构
- 系统输出可直接复制使用的成品 Prompt

如果用户已经有 Prompt，则产品进一步支持：

- 识别 Prompt 为什么不好
- 判断缺失了哪类控制层
- 给出最小必要修复

如果用户已有输出结果，则产品进一步支持：

- 判断这个输出质量如何
- 找出主要缺陷
- 反推需要补哪类 Prompt 结构

## 三 产品愿景

让用户不需要学习复杂的 Prompt Engineering 方法论，也不需要手工拼装复杂 Prompt，只需要表达自己的需求，系统就能稳定生成高质量 Prompt 成品。

更进一步地说，产品愿景不是让用户学会说专业语言，而是让系统替用户补上那部分原本只有专业人士才能补上的结构、术语、边界和表达精度，从而尽可能减小普通用户与专业领域之间的沟通摩擦。

长期来看，Prompt Studio 的目标不是做“更多 Prompt 模板”，而是成为一个真正的 Prompt Operating System：

- 理解需求
- 组装 Prompt
- 调试 Prompt
- 评估 Prompt
- 沉淀 Prompt 资产

## 四 产品目标

### 目标 1

让普通用户只输入一句需求，也能获得高质量 Prompt。

这里的关键不只是“输入短”，而是“输入可以不专业、不精确、不完整”。产品必须能够容忍这种普通用户表达方式，并把它翻译成更适合与 LLM 协作的输入结构。

### 目标 2

让进阶用户在 Prompt 不理想时，不再依赖盲改，而能系统化调试。

### 目标 3

让 Prompt 的生成、调试、评估形成闭环，而不是一次性文本生产。

### 目标 4

为未来的团队协作、Prompt 资产管理和工作流集成提供基础架构。

## 五 非目标

以下不属于 MVP 或当前版本的目标：

- 直接替用户完成最终任务本身
- 做成通用聊天机器人
- 做成 Prompt 模板 marketplace
- 做复杂的协作权限系统
- 做多模型 benchmark 平台
- 做复杂可视化工作流编排器
- 绑定任何特定垂直行业语境

## 六 目标用户

### 用户类型 A：轻度 Prompt 用户

特征：

- 经常使用 AI，但不会系统写 Prompt
- 希望快速拿到可直接使用的 Prompt
- 不想学习复杂 Prompt 方法论
- 面对陌生领域时，不知道如何使用专业术语准确表达需求

核心需求：

- 一句话需求 -> 成品 Prompt
- 模糊表达 -> 更专业、更清晰、更高质量的 Prompt

### 用户类型 B：进阶 Prompt 用户

特征：

- 已经能写一些 Prompt
- 但结果不稳定
- 不知道 Prompt 为什么有时好有时差

核心需求：

- 识别 Prompt 质量问题
- 调试已有 Prompt
- 建立更稳定的 Prompt 结构

### 用户类型 C：系统集成者 / 团队使用者

特征：

- 希望把 Prompt 生成能力嵌入工具、产品或团队流程
- 关注输入输出契约和系统边界

核心需求：

- 稳定 API
- 明确行为模式
- 可复用 skill / 模块化结构

## 七 核心用户问题

Prompt Studio 要解决的不是“模型不会回答”，而是“用户无法稳定得到高质量回答所对应的高质量 Prompt”。

更具体地说，它解决五类问题：

### 问题 1：用户需求很短，但想要高质量结果

用户只会说“帮我分析这个问题”，却无法补足 Prompt 所需结构。

更常见的情况是，用户甚至不知道某个陌生领域里应该问什么、用什么词、强调什么边界，只能用生活化语言、模糊意图或者不准确术语表达。结果不是用户没有需求，而是需求没有被翻译成模型可高质量处理的形式。

### 问题 2：Prompt 能用，但经常输出普通结果

输出常见问题包括：

- 表层复述
- 模板腔
- 正确废话
- 伪深度
- 无批判赞美
- 不可执行

### 问题 3：用户不会调 Prompt

一旦结果不好，大多数用户只能重写或乱改 Prompt，而无法定位缺失结构。

### 问题 4：Prompt 质量没有统一判断标准

用户通常只能主观感觉“这次好像不错”，但没有稳定评估框架。

### 问题 5：Prompt 经验难以沉淀

即使偶尔写出好 Prompt，也无法沉淀成可复用资产。

## 八 核心产品能力

MVP 先聚焦三种核心模式。

但在这三种模式之下，必须始终内置一个更底层的核心能力：

**需求翻译与专业表达补全能力。**

也就是说，无论是 Generate、Debug 还是 Evaluate，系统都应默认承担一部分“替用户把话说明白、说专业、说结构化”的责任。

### 1. Generate

输入一句需求，输出成品 Prompt。

### 2. Debug

输入原始任务、当前 Prompt 和当前输出，输出修复后的 Prompt。

### 3. Evaluate

输入 Prompt 或输出文本，输出结构化质量评估结果。

这三种模式共同构成产品最小闭环。

## 九 MVP 范围

### MVP 必做

- 一句话需求 -> 成品 Prompt
- Prompt 调试
- Prompt 评估
- 一键复制 Prompt
- 任务类型自动识别
- 基础失败模式识别
- 控制模块自动装配

### MVP 可选

- 简单历史记录
- 输出偏好开关
- Prompt-only 输出模式

### MVP 不做

- Prompt 模板市场
- 团队协作
- 多人资产管理
- 高级模块可视化编辑器
- 多模型对比平台
- 大规模社区内容系统

## 十 核心体验原则

### 原则 1：一句需求优先

用户必须能在最短输入路径下获得价值。

并且这里的一句话不要求专业、完整或严谨，系统应默认用户表达是粗糙的。

### 原则 2：默认先给成品 Prompt

解释和诊断可以有，但成品 Prompt 必须是默认主输出。

### 原则 3：复杂性后台化

任务识别、模块装配、失败模式路由等复杂逻辑不应要求用户显式操作。

同样，专业术语翻译、任务结构补全和表达精炼，也应尽量由系统在后台完成，而不是强迫用户先变专业。

### 原则 4：调试比模板更重要

产品的差异化来自“修 Prompt 的能力”，而不是“模板数量”。

### 原则 5：结果必须可复制使用

任何主要输出都必须能直接复制，而不是停留在说明层。

## 十一 MVP 信息架构

```text
Prompt Studio
├── Generate
├── Debug
├── Evaluate
└── History（可选）
```

### Generate 页面

输入：

- 一句话需求

可选项：

- 显示诊断摘要
- 输出偏好：深度 / 可执行性 / 自然文风

输出：

- 诊断摘要
- 最终 Prompt
- 一键复制

### Debug 页面

输入：

- 原始任务
- 当前 Prompt
- 当前输出

输出：

- 失败模式
- 缺失控制层
- 最小必要修复
- 修复后的 Prompt

### Evaluate 页面

输入：

- 一个 Prompt
- 或一个输出文本

输出：

- 分项评分
- 总分
- 最主要缺陷
- 最优先补层

## 十二 核心系统逻辑

Prompt Studio 的后台逻辑应遵循以下流程：

```text
输入一句需求
-> 识别任务类型
-> 识别输出产物
-> 识别质量目标
-> 预判失败模式
-> 加载默认控制模块
-> 根据质量目标做加权调整
-> 装配五层 Prompt 结构
-> 输出成品 Prompt
```

对于 Debug 模式：

```text
输入原始任务 + 当前 Prompt + 当前输出
-> 识别失败模式
-> 判断缺失控制层
-> 给出最小必要修复
-> 输出修复后的 Prompt
```

对于 Evaluate 模式：

```text
输入 Prompt 或输出
-> 按统一评估维度打分
-> 输出主要缺陷
-> 输出建议补层
```

## 十四 成功指标

MVP 阶段建议关注以下指标：

### 指标 1：首轮成品 Prompt 可用率

用户首次生成后，不做修改即可直接使用的比例。

### 指标 2：Debug 后提升率

调试后 Prompt 明显优于初版 Prompt 的比例。

### 指标 3：复制使用率

生成结果被实际复制的比例。

### 指标 4：模式分布

Generate / Debug / Evaluate 三模式的使用占比。

### 指标 5：留存中的高频场景

用户最常用的任务类型分布，便于后续做预设和优化。

### 指标 6：低质量输入转高质量 Prompt 的成功率

即使用户输入模糊、口语化、缺少专业术语，系统仍能产出可直接使用高质量 Prompt 的比例。

## 十五 风险与应对

### 风险 1：产品退化成模板库

应对：坚持“生成 + 调试 + 评估”闭环，不只堆模板。

### 风险 2：前台暴露太多复杂结构

应对：默认隐藏模块和工程细节，仅在高级模式中逐步开放。

### 风险 3：输出解释过多，成品 Prompt 不够突出

应对：默认把成品 Prompt 作为主输出区。

### 风险 4：用户以为这是聊天产品

应对：产品文案和交互都要强化“编译 Prompt”而不是“直接回答问题”。

## 十八 最终判断

Prompt Studio 已经具备独立产品化的条件。

它不再只是一个文档工程，也不再只是提示词经验沉淀，而已经具备一个产品最关键的三点：

- 明确对象：一句需求 -> 成品 Prompt
- 明确流程：生成 / 调试 / 评估
- 明确结构：控制模块 + 五层 Prompt 装配

因此，从现在开始，它应被视为一个独立 Prompt 产品，而不是方法论文档的延伸。

它最终要解决的核心问题不是“再做一个 Prompt 工具”，而是：帮助普通用户跨越专业表达门槛，让他们即使不会用专业术语、不会精炼定义问题，也能通过产品获得更高质量的 Prompt，进而获得更高质量的 LLM 响应。
