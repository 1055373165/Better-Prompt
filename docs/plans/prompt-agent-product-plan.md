# Prompt Agent Product Plan

## 一 产品定义

产品暂定名：Prompt Agent

Prompt Agent 是一个面向个人用户的高质量 Prompt 优化 agent，而不是单纯的 Prompt 生成器。用户输入一段话后，系统允许用户明确选择以下三类核心任务：

- 生成更优质的系统提示词
- 调试提示词，分析优劣势并给出修复方案
- 评估提示词，对 Prompt 质量进行结构化打分

在此基础上，Prompt Agent 不停留在“一次输出一个结果”，而是进一步支持一个闭环：

- 用户先生成 Prompt
- 再对 Prompt 做调试
- 再做评估
- 再由 agent 基于评估结果继续优化

因此，这个产品本质上不是一个 Prompt 页面，而是一个围绕 Prompt 生命周期工作的个人 agent。

更准确地说，这个产品首先不是为 Prompt 专家设计的，而是为那些表达能力有限、缺乏专业术语、面对陌生领域时不知道如何清晰描述需求的普通用户设计的。用户真正的痛点并不是“他不会点按钮”，而是“他不知道怎样把脑中的模糊意图翻译成模型真正能高质量理解的输入”。Prompt Agent 的任务，就是替用户完成这层需求翻译、结构补全和专业语义补足，尽可能减小普通用户与专业领域之间的表达摩擦。

## 二 产品定位

Prompt Agent 的定位应当是：

**一个帮助个人用户持续生成、分析、评估并优化 Prompt 的个人 Prompt Agent。**

与模板库相比，它更强调动态生成。

与普通 AI 聊天工具相比，它更强调目标明确的 Prompt 生命周期管理。

与纯 Prompt 编译器相比，它更进一步，因为它不只负责生成，还负责自我诊断和继续优化。

从用户价值层看，它更像一个“需求翻译 agent”。用户给出的可能只是模糊意图、生活化语言、错误术语甚至半句描述，而系统要负责把这些不完整表达翻译成更接近专业领域问题定义的 Prompt 结构。

## 三 用户价值

### 价值一

用户不必懂 Prompt Engineering，也能得到高质量系统提示词。

### 价值二

用户不必靠感觉修改 Prompt，系统可以主动指出 Prompt 的结构问题。

### 价值三

用户可以明确知道一个 Prompt 为什么强、为什么弱，而不是只看结果好不好。

### 价值四

用户可以把“生成 -> 调试 -> 评估 -> 再优化”形成闭环，而不是每次从零开始。

### 价值五

用户即使不掌握专业术语、不熟悉陌生领域，也能借助 agent 获得更接近专家表达水平的 Prompt，从而得到更高质量的 LLM 响应。

## 四 用户核心交互

产品最核心的交互非常简单。

### 用户输入

用户输入一段话，可以是：

- 我想做什么
- 我的原始需求
- 我的 Prompt 初稿
- 我当前的 Prompt 和输出

### 用户选择模式

用户在输入后选择以下三种模式之一：

#### 1. Generate

目标：生成更优质的系统提示词

#### 2. Debug

目标：分析一个 Prompt 的优劣势，并给出最小必要修复方案

#### 3. Evaluate

目标：对 Prompt 质量进行结构化评估与打分

### 用户继续优化

当系统返回结果后，用户可点击继续优化，让 agent 基于前一步结果继续迭代。

这意味着产品会从“单次工具”升级成“连续优化 agent”。

## 五 核心产品闭环

Prompt Agent 的核心闭环如下：

```text
用户输入一段话
-> 选择模式 Generate / Debug / Evaluate
-> Agent 产出结果
-> 用户查看结果
-> 用户选择继续优化
-> Agent 基于已有上下文继续生成修复或增强结果
-> 形成 Prompt 优化循环
```

这个闭环是产品差异化的关键。因为真正高质量的 Prompt 往往不是一次生成出来的，而是在生成、诊断和修复中逐渐成形。

更重要的是，这个闭环不是只为 Prompt 爱好者设计，而是为了弥补普通用户在“需求表达”和“专业语义组织”上的天然劣势。Agent 必须替用户多做一步：把用户原始表达补全成更适合被模型理解和执行的任务结构。

## 六 三种核心模式定义

### 1. Generate 模式

#### 用户目标

给定一句需求，让系统输出一份更优质的系统提示词。

#### 输入

- 一句话需求
- 或一段任务描述

#### 输出

- 任务诊断摘要
- 高质量系统提示词
- 采用的关键优化策略

#### 用户收益

用户不需要自己手写系统提示词，只需要表达任务目标。

### 2. Debug 模式

#### 用户目标

分析一个已有 Prompt 的优缺点，并给出修复方案。

#### 输入

- 原始任务
- 当前 Prompt
- 可选：当前输出

#### 输出

- Prompt 优势
- Prompt 弱点
- 高风险失败模式
- 缺失控制层
- 最小必要修复
- 修复后的 Prompt

#### 用户收益

用户不再凭感觉改 Prompt，而能系统化理解 Prompt 问题。

### 3. Evaluate 模式

#### 用户目标

判断一个 Prompt 是否足够强，并明确问题在哪里。

#### 输入

- Prompt 文本
- 可选：输出文本

#### 输出

- 多维评分
- 总分
- 最主要缺陷
- 优先修复建议

#### 用户收益

用户获得可重复使用的评估标准，而不是主观印象。

## 七 Continue Optimization 机制

Prompt Agent 之所以是 agent，而不只是工具，关键在于它必须支持继续优化。

在每种模式结果返回后，都要允许用户继续推进。

### 在 Generate 后

用户可以要求：

- 再增强深度
- 再增强可执行性
- 再降低 AI 味
- 改成更适合某类任务

### 在 Debug 后

用户可以要求：

- 再修一次
- 更激进地优化
- 保持原意但增强结构
- 针对某个问题单独修复

### 在 Evaluate 后

用户可以要求：

- 根据评分结果自动优化
- 只修最低分项
- 重新生成一个更优版本

这样，产品体验会从“你拿一个结果走人”，升级为“你和 agent 一起优化到更强结果”。

## 八 MVP 范围

### MVP 必做

- 三种模式：Generate / Debug / Evaluate
- 单轮输入输出
- Continue Optimization 入口
- 结构化结果展示
- 一键复制 Prompt
- 历史上下文仅保留单会话内链路
- 需求翻译与结构补全能力
- 面向普通用户的低门槛输入体验

### MVP 可选

- 多轮优化历史追踪
- Prompt 版本对比
- 偏好预设

### MVP 不做

- 团队协作
- Prompt 市场
- 多人资产管理
- 工作流编排器
- 多模型 benchmark 平台

## 九 用户界面建议

产品核心页面建议采用单页面 + 模式切换结构：

```text
Prompt Agent
├── Input Area
├── Mode Selector
│   ├── Generate
│   ├── Debug
│   └── Evaluate
├── Result Area
└── Continue Optimization Actions
```

结果区应根据模式动态展示不同模块，但视觉风格应统一，保持 agent 工具的连贯感。

在交互文案上，产品不应默认用户懂 Prompt Engineering。系统提示语、空状态和操作说明都应使用面向普通用户的语言，例如强调“描述你想做什么”，而不是“请输入高质量 Prompt 需求”。

## 十 差异化能力

Prompt Agent 要避免沦为另一个“输入一句话生成 Prompt”的浅层产品。它的真正差异化在于以下三点：

### 差异化一：三模式闭环

不是只会生成，还会调试和评估。

### 差异化二：控制结构驱动

不是机械改写，而是通过问题重定义、认知路径、质量门槛和模块装配来优化 Prompt。

### 差异化三：Continue Optimization

不是一次性结果，而是 Prompt 生命周期中的持续优化 agent。

### 差异化四：需求翻译能力

产品不是等用户先学会专业表达，而是主动吸收用户的模糊输入、口语表达和非专业描述，替用户补足专业领域所需的问题定义、约束和表达结构。

## 十一 产品成功标准

### 指标 1

用户首次输入一句需求后，能拿到可直接使用 Prompt 的比例。

### 指标 2

Debug 后 Prompt 被认为优于原始版本的比例。

### 指标 3

Evaluate 后用户继续触发优化的比例。

### 指标 4

用户复制最终 Prompt 的比例。

### 指标 5

用户在单会话内多轮优化的比例。

## 十二 风险与控制

### 风险 1

产品被用户误解成聊天机器人。

控制方式：

- 界面与文案强调 `Prompt Agent`
- 输出始终围绕 Prompt，而不是直接任务答案

### 风险 2

Generate 模式掩盖 Debug / Evaluate 价值。

控制方式：

- 在结果页显式展示“继续调试”“继续评估”“继续优化”入口

### 风险 3

产品被做成模板库。

控制方式：

- 强调按用户输入动态生成
- 后台显式使用控制模块与失败模式路由

## 十三 下一步

在产品方案之后，最合理的下一步是：

1. 定义 Agent 架构
2. 定义 API 契约
3. 定义前端交互规格
4. 落地 MVP 骨架

## 十四 最终判断

Prompt Agent 应被定义为一个个人高质量 Prompt 优化 agent，而不是 Prompt 页面、Prompt 模板库或普通 AI 聊天工具。

只要产品始终围绕 `Generate / Debug / Evaluate / Continue Optimization` 四个核心动作设计，它就会具备明确定位、强闭环和清晰差异化。

而从更底层的产品价值看，它的真正使命是：让普通用户即使不具备专业表达能力，也能借助 agent 跨越陌生领域的语言门槛，获得更高质量的 Prompt，进而获得更高质量的 LLM 响应。
