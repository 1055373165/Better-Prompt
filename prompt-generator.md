
# Prompt Generator

这份文档是整套 Prompt System 的最终用户入口。它不要求用户理解 Prompt Playbook、控制模块、任务路由或失败模式词典。它只做一件事：让用户输入一句需求、一个问题、一个模糊目标，系统就输出一份可以直接复制使用的成品提示词。

如果把前面的文档分别理解为方法论、速查表、样例库和控制模块层，那么这份文档就是面向最终用户的“终极入口”。正常情况下，用户只需要使用这里的模板，而不需要再翻阅其他文档。

## 零 单文件终极模板

如果你只想复制一段母提示词长期使用，那么直接复制下面这一段就够了。它是这套体系压缩后的最终版，目标就是让用户只输入一句需求，系统直接输出成品提示词。

```text
我会给你一句需求、一个问题，或者一个模糊目标。

你的任务不是直接回答，而是先把它编译成一份可以直接使用的高质量 Prompt。

你必须自动完成以下工作：
1. 识别任务类型
2. 识别用户真正想要的输出产物
3. 识别用户最在意的质量目标
4. 识别这个任务最容易出现的低质量输出
5. 自动补足最合适的角色定义、问题重定义、认知路径、质量门槛和文风约束
6. 输出一份可以直接复制使用的最终 Prompt

请优先遵守以下原则：
- 用户的描述是起点，不是边界
- 不要停留在表层复述，先识别问题本质
- 不要输出模板腔、正确废话或伪深度
- 如果任务需要判断或建议，必须补足边界、失效条件和可执行性
- 正文风格应自然、连贯、信息密度高

输出格式必须严格如下：

诊断摘要：
- 任务类型：
- 输出产物：
- 质量目标：
- 高风险失败模式：

可直接使用的最终 Prompt：
【在这里输出完整成品 Prompt】

要求：
- 最终 Prompt 必须完整、可直接使用，不要输出半成品
- 不要只解释思路，必须真正产出 Prompt
- 如果用户信息不足，允许你根据任务类型自动补足最常见且最有价值的控制约束
- 优先防止表层复述、模板腔、正确废话、伪深度和不可执行

我的需求是：
【把一句需求写在这里】
```

这就是最推荐的最终形态。今后如果你只保留一个模板，保留它就够了。

## 一 最简使用说明

使用方法非常简单。

第一步，写下一句你的真实需求。
第二步，把它放进下面的生成器模板。
第三步，拿到系统输出的“最终 Prompt”。
第四步，直接复制那个最终 Prompt 去使用。

你不需要提前说明任务类型，也不需要理解什么叫问题重定义、认知路径或质量门槛。生成器会自动判断这些东西，并自动补齐高质量输出所需的控制结构。

如果你不想读后面的说明，可以直接跳过全文，只使用上面的“单文件终极模板”。

## 三 极简版入口模板

如果你觉得上面的模板仍然有点长，那么下面这份是最短版。它更适合日常高频调用。

```text
把我下面这句话，转换成一份可以直接使用的高质量 Prompt。

你要自动判断任务类型、输出目标、质量目标和高风险失败模式，并补足角色定义、问题重定义、认知路径、质量门槛和文风约束。

最后只输出两部分：
1. 诊断摘要
2. 可直接复制使用的最终 Prompt

我的需求是：
【把一句需求写在这里】
```

## 四 只输出成品 Prompt 的纯净模式

有些时候你不想看诊断摘要，只想拿到最终 Prompt。那就用下面这个版本。

```text
我会给你一句需求。

不要直接回答这个需求，而是把它编译成一份可以直接使用的高质量 Prompt。

你需要自动判断任务类型，并补足高质量 Prompt 所需的结构。

不要输出分析过程，不要输出诊断摘要，只输出最终成品 Prompt。

我的需求是：
【把一句需求写在这里】
```

这个模式最适合已经信任系统、只想快速拿成品的情况。

## 五 输出多个候选版本的比较模式

如果你想做 Prompt 对比实验，可以使用下面这个版本。它会一次输出多个候选 Prompt，便于你做 A/B 测试。

```text
我会给你一句需求。

请不要直接回答需求，而是基于它生成 3 个可直接使用的高质量 Prompt 候选版本。

要求：
1. 三个版本都必须能直接使用
2. 三个版本的差异必须明确
3. 一个版本偏重深度
4. 一个版本偏重可执行性
5. 一个版本偏重自然文风与可读性
6. 每个版本前用一句话说明它最适合什么场景

我的需求是：
【把一句需求写在这里】
```

这个模式特别适合当你还不确定自己更需要“深度”“行动性”还是“风格自然度”时使用。

## 六 输出带调试建议的增强模式

如果你希望系统不仅给出成品 Prompt，还提前告诉你未来可能怎么修，可以使用下面这个版本。

```text
我会给你一句需求。

你的任务不是直接回答，而是生成一份可直接使用的高质量 Prompt，并附带一份极简调试建议。

输出格式：
1. 诊断摘要
2. 可直接使用的最终 Prompt
3. 如果未来输出不理想，优先应补哪一层约束

我的需求是：
【把一句需求写在这里】
```

这个模式的好处是，用户第一次拿到 Prompt 时，就知道如果结果偏差，应该往哪里修，而不是重新开始。

## 七 推荐默认设置

如果你不知道该用哪一个版本，建议默认使用“最终用户入口模板”。因为它在透明度和可用性之间最平衡。

如果你是高频重度用户，建议默认使用“极简版入口模板”。

如果你在做实验和对比，建议用“多个候选版本的比较模式”。

如果你要把这套系统嵌入产品或工具，建议默认使用“只输出成品 Prompt 的纯净模式”，这样用户体验最简洁。

## 九 给系统集成者的最短说明

如果你以后要把这份文档接入应用、网页、插件或自动化工具，那么对外只需要暴露一个输入框：

- 用户输入一句需求
- 系统内部调用这个生成器模板
- 返回“最终 Prompt”或“诊断摘要 + 最终 Prompt”

对用户来说，这就是一个 Prompt 编译器。用户给的是自然语言需求，系统返还的是可直接工作的 Prompt 成品。

## 十 最后原则

整套体系最终收敛到这里以后，最重要的原则只有一条：不要让用户学习复杂方法，而要让系统承担复杂性。

真正成熟的 Prompt System，不是让用户理解更多，而是让用户输入更少，却得到更强的结果。`prompt-generator.md` 的意义，就是把这种体验真正固定下来。
